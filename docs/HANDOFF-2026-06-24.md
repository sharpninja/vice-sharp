## Diagnosed: "89% but plays 300-500% fast" capture bug - RESOLVED

Root cause was NOT the gate. `FfmpegVideoRecorder` used `dropWhenFull` on the video writer, and the
ffmpeg input used synthetic `1/fps` PTS, so dropped frames compressed the clip timeline. Fix
(`b218e28`): wall-clock input PTS + CFR output duplicate-fills gaps. Confirmed by `ffprobe` on a
fresh clip: CFR 50.12, `DROP 0`, duration matches frame count. Fast-playback gone.

## Open follow-up: audio-on emulation pace ~77% (single-thread bound)

Live measurement on the running app (idle C64 BASIC):
- Audio OFF (`VICESHARP_AUDIO=0`): **94% / 51.6 fps** (~real time).
- Audio ON: **77% / ~39 fps**, ViceSharp at ~120% of ONE core (15 of 16 cores idle).

So the ~77% ceiling is **SID emulation CPU cost on the single emulation thread**, not the gate
(the deficit fix already took audio-on from 19% -> 77%). The earlier "0-40% fluctuating" reading
was MY background AOT build + MSBuild nodes contending for cores during the user's test window;
machine is clean now.

Target for optimization: `src/ViceSharp.Chips/Audio/Sid6581.cs` `Tick()` (line ~479), called every
phi2 cycle (~985k/s). It runs the 3-voice accumulator + `ProcessEnvelope` x3 + noise/sync logic per
cycle; `GenerateSample()` (line ~90) runs per output sample (44.1 kHz). The per-cycle envelope work
is the prime suspect. Approaches to try: skip/short-circuit `ProcessEnvelope` when a voice's
envelope is idle, batch-clock the SID between samples, or reduce per-cycle float work. Profile first
(no `dotnet-trace` on the AOT exe; bisect or add timing counters). The user opted "Both, fixes
first" - the two UI/overlay fixes are done; SID perf is the remaining item.
